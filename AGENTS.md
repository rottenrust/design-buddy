# AGENTS.md

## Перед началом

1. Прочитай `docs/MVP.md` целиком.
2. Считай его единственным источником продуктовых и технических требований.
3. Текущий код — рабочая база v0.1. Дорабатывай его до v0.2 поверх существующей реализации.
4. Не очищай репозиторий и не возвращай удалённую архитектуру из старых веток или PR без отдельного указания.
5. Всегда общайся с владельцем проекта на русском языке: планы, промежуточные сообщения, финальные ответы и пояснения должны быть на русском.

## Цель проекта

Сделать простого персонального Telegram-наставника для начинающего UX/UI-дизайнера с:

- естественным диалогом;
- постоянной персональной памятью в SQLite;
- первичным опросом по опыту и навыкам;
- структурированным профилем основных UX-навыков;
- одним практическим заданием на каждый основной навык;
- пошаговыми объяснениями и проверкой по рубрике;
- отдельным запуском агента на каждое сообщение;
- восстановлением контекста после перезапуска.

## Главный продуктовый принцип

Бот не должен быть обычным ChatGPT с промптом про дизайн.

Основной цикл v0.2:

```text
опрос → профиль навыков → конкретное задание → объяснение и практика
→ проверка по рубрике → обновление профиля
```

Свободный диалог остаётся, но активное задание должно вести дизайнера по управляемому учебному сценарию.

## Обязательный стек

- Python 3.12
- aiogram 3
- SQLAlchemy 2
- aiosqlite и SQLite
- Pydantic и pydantic-settings
- установленный и заранее авторизованный agent CLI
- pytest, pytest-asyncio, ruff

Бот не должен хранить API-ключ модели или напрямую вызывать API LLM-провайдера. Авторизация агента выполняется вне приложения, в среде его запуска.

Приложение при каждом запуске агента передаёт в `codex exec` модель `gpt-5.4-mini` и `medium` reasoning effort. Эти параметры фиксируются в коде и не хранятся в `.env`.

## Архитектурные ограничения

Выбирай простой понятный код вместо универсальной архитектуры.

Без отдельного запроса не добавляй:

- AgentBridge;
- SessionManager;
- универсальный ProgramEngine;
- ContextBuilder framework;
- planner, evaluator или несколько агентов;
- event bus, workflow engine или tool registry;
- PostgreSQL, Redis, Celery, FastAPI;
- векторную базу, RAG или embeddings;
- прямую интеграцию с API LLM-провайдера;
- постоянную или resume-сессию агента;
- автоматическое завершение заданий без подтверждения пользователя.

Допустимы небольшие сервисы, напрямую соответствующие:

- каталогу навыков и заданий;
- первичному опросу;
- профилю навыков;
- пошаговой работе над активным заданием;
- постоянной памяти.

## Границы состояния

- Источник истины — локальная SQLite-база и `data/tasks.yaml`.
- Вся история не передаётся агенту.
- Agent-сессия не используется как память.
- Персональная память, профиль навыков и контекст разговора — разные сущности.
- Профиль навыков нельзя хранить только в произвольных `memory_items`.
- Агент может предложить обновление памяти, профиля или завершение задания, но backend валидирует результат.
- `skill_id` должен существовать в каталоге.
- Самооценка не считается подтверждённым уровнем без evidence.
- Агент не меняет `active_task_id` и не переводит задание в `done` без явного подтверждения дизайнера.
- Неизвестные поля структурированного ответа отклоняются.
- Каждый входящий запрос запускает новый независимый процесс агента без resume существующей сессии.

## Правила работы с текущим кодом

- Не переписывай работающий вертикальный сценарий без необходимости.
- Сохрани Telegram, SQLite, one-shot runner, сообщения, rolling summary, память и текущий прогресс заданий.
- Изменения схемы БД должны сохранять существующие данные и не требовать ручного удаления файла базы.
- Расширяй существующие модели и сервисы либо добавляй маленькие специализированные сервисы.
- Не создавай абстракции только ради будущего расширения.
- Не меняй модель, reasoning effort и способ запуска агента, если задача этого явно не требует.

## Порядок реализации v0.2

1. Расширить `data/tasks.yaml`: 12 навыков и одно задание на каждый навык.
2. Обновить Pydantic-модели каталога и его валидацию.
3. Добавить постоянный профиль навыков и безопасное обновление SQLite.
4. Добавить состояние и сценарий первичного опроса.
5. Включить профиль и полное описание задания в prompt.
6. Обновить системный prompt под наставническое поведение.
7. Реализовать пошаговый сценарий задания и проверку по рубрике.
8. Добавить просмотр профиля и рекомендацию следующего задания.
9. Обновить тесты и README.

Следуй этому порядку. Не реализуй следующие этапы через временные заглушки, которые ломают предыдущие.

## Поведение агента для активного задания

Агент обязан:

- объяснить, какой навык тренируется и зачем;
- назвать ожидаемый результат;
- давать один небольшой шаг за раз;
- при непонимании дать простое объяснение, пример и микро-шаг;
- не выполнять основную работу вместо дизайнера;
- проверять результат по рубрике;
- конкретно отмечать сильные стороны и доработки;
- обновлять профиль навыка только на основании наблюдаемого evidence.

## Качество

- Новая логика покрывается тестами.
- Существующие тесты не удаляются без обоснования.
- В тестах используется fake agent runner; запуск реального агента и сетевые вызовы запрещены.
- Перед завершением задачи запускаются:

```bash
ruff check .
pytest
```

- Секреты и токены не коммитятся.
- Один сбой агента не должен останавливать polling.
- Не выполняй merge: слияние делает только владелец репозитория.
