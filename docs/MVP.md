# Design Buddy v0.2

## 1. Цель

Создать персонального Telegram-наставника для начинающего UX/UI-дизайнера, который:

- ведёт естественный диалог о UX/UI, портфолио и профессиональном развитии;
- хранит постоянную персональную память между сообщениями и перезапусками;
- сначала формирует профиль дизайнера через короткий опрос по опыту и навыкам;
- содержит карту основных UX-навыков и по одному практическому заданию на каждый навык;
- объясняет, что, как и зачем делать в рамках активного задания;
- проверяет результат по заранее заданным критериям;
- обновляет профиль навыков на основании ответов и выполненных работ;
- запускает агента отдельно для каждого сообщения и не держит постоянную agent-сессию.

Это персональный наставник, а не образовательная платформа и не универсальный агентный фреймворк.

## 2. Основа и изменение в v0.2

Текущий MVP уже реализует:

- Telegram-бота;
- SQLite;
- one-shot запуск `codex exec` на каждое сообщение;
- историю сообщений и rolling summary;
- произвольную долговременную память;
- выбор активного задания из YAML;
- сохранение прогресса задания.

v0.2 дорабатывается поверх текущей реализации без очистки репозитория и без возврата удалённой сложной архитектуры.

Главное изменение:

```text
первичный опрос
→ постоянный профиль дизайнера
→ карта навыков
→ задание на конкретный навык
→ пошаговое объяснение и практика
→ проверка по рубрике
→ обновление профиля навыков
```

Свободный диалог сохраняется, но активное задание должно запускать управляемый учебный сценарий, а не обычный разговор с дизайн-промптом.

## 3. Основной технический сценарий

Для каждого входящего сообщения backend:

1. Проверяет Telegram ID.
2. Находит или создаёт пользователя.
3. Загружает из SQLite:
   - персональную память;
   - профиль навыков;
   - состояние первичного опроса;
   - активное задание и его полное описание;
   - прогресс задания;
   - rolling summary;
   - последние сообщения.
4. Собирает ограниченный контекст для агента.
5. Запускает новый независимый процесс установленного agent CLI.
6. Получает и валидирует структурированный результат.
7. Отправляет пользователю только текст ответа.
8. Сохраняет сообщения и разрешённые изменения состояния.

```text
Telegram → SQLite context → one agent run → Telegram reply → persist result
```

Постоянная agent-сессия не используется. Источник истины — SQLite и структурированный каталог навыков и заданий.

## 4. Первичный опрос

При первом `/start` бот не предлагает свободный чат или случайное задание. Он запускает короткий последовательный опрос.

Цели опроса:

- узнать профессиональную цель;
- понять опыт и существующие проекты;
- определить используемые инструменты;
- выяснить предпочитаемый формат объяснений и темп;
- получить первичную самооценку основных UX-навыков;
- проверить часть навыков короткими ситуационными вопросами;
- сформировать начальный профиль навыков.

Правила опроса:

- один основной вопрос за сообщение;
- вопросы задаются естественным языком;
- это не экзамен и не тест с баллами;
- агент просит привести конкретный пример, если ответ слишком общий;
- самооценка не считается подтверждённым уровнем;
- после опроса бот показывает краткое резюме сильных сторон и зон развития;
- дизайнер может исправить выводы до завершения опроса.

Статусы:

```text
not_started
in_progress
completed
```

После завершения опроса профиль должен восстанавливаться после перезапуска приложения.

## 5. Уровни навыка

Использовать четыре понятных уровня:

```text
unknown      — данных пока недостаточно
awareness    — понимает назначение и основные термины
guided       — выполняет с пошаговой помощью
independent  — выполняет и объясняет решения самостоятельно
```

Для каждого навыка хранить отдельно:

- `self_level` — самооценка дизайнера;
- `assessed_level` — уровень, подтверждённый ответами и выполненными заданиями;
- `confidence` — уверенность оценки: `low | medium | high`;
- `evidence` — краткое основание оценки;
- `strengths` — подтверждённые сильные стороны;
- `gaps` — конкретные пробелы и зоны развития.

Не использовать проценты и псевдоточные числовые оценки.

## 6. Карта основных навыков

В v0.2 должен быть зафиксирован следующий минимальный набор:

1. `problem_framing` — формулирование пользовательской проблемы.
2. `research_planning` — планирование исследования.
3. `user_interviews` — подготовка и проведение пользовательских интервью.
4. `research_synthesis` — синтез наблюдений и формулирование инсайтов.
5. `user_needs` — описание потребностей и контекста пользователя на основе данных.
6. `user_flows` — task analysis и пользовательские сценарии.
7. `information_architecture` — структура контента и навигации.
8. `interaction_design` — проектирование взаимодействия и состояний интерфейса.
9. `prototyping` — создание проверяемых прототипов.
10. `usability_testing` — подготовка, проведение и разбор юзабилити-тестов.
11. `accessibility` — базовая доступность и инклюзивность интерфейса.
12. `design_rationale` — аргументация решений и итерация по обратной связи.

На каждый навык в каталоге должно приходиться ровно одно основное практическое задание v0.2.

## 7. Каталог навыков и заданий

Каталог хранится в `data/tasks.yaml` и загружается при запуске.

Рекомендуемая структура:

```yaml
skills:
  - id: problem_framing
    order: 10
    title: Формулирование пользовательской проблемы
    description: ...
    diagnostic_questions:
      - ...
      - ...

tasks:
  - id: problem_framing_task
    skill_id: problem_framing
    order: 10
    title: Сформулировать пользовательскую проблему
    why_it_matters: ...
    context: ...
    brief: ...
    deliverables:
      - ...
    learning_steps:
      - ...
    explanation_topics:
      - ...
    rubric:
      - id: user
        title: Понятно, о каком пользователе идёт речь
      - id: context
        title: Описан контекст использования
    common_mistakes:
      - ...
    done_definition: ...
```

Обязательные поля задания:

```text
id
skill_id
order
title
why_it_matters
context
brief
deliverables
learning_steps
explanation_topics
rubric
common_mistakes
done_definition
```

Backend валидирует:

- уникальность `skill.id` и `task.id`;
- существование `skill_id` у каждого задания;
- наличие ровно одного основного задания на каждый навык;
- непустую рубрику и список результатов;
- порядок отображения.

## 8. Учебный сценарий задания

После выбора задания бот не должен отвечать только «можем продолжать» и ждать нового промпта от пользователя.

Он обязан:

1. Назвать навык и объяснить, зачем он нужен.
2. Кратко описать итоговый результат задания.
3. Уточнить исходный опыт, если он неизвестен.
4. Дать только первый небольшой шаг.
5. Объяснять теорию в объёме, необходимом для текущего шага.
6. При непонимании дать:
   - простое объяснение;
   - короткий пример;
   - микро-задание или следующий конкретный вопрос.
7. Не выполнять основную работу вместо дизайнера.
8. Проверять результат по рубрике задания.
9. Называть конкретно:
   - что уже получилось;
   - что требует доработки;
   - какой один следующий шаг сделать.
10. Предлагать завершение только после прохождения обязательных критериев.
11. После завершения обновлять связанный профиль навыка.

Агент может адаптировать объяснение и размер шага, но не может менять цель навыка, рубрику или обязательные результаты задания.

## 9. Постоянная память

Память состоит из трёх разных частей.

### 9.1. Персональная память

Хранит устойчивые факты:

- профессиональная цель;
- опыт;
- текущие проекты;
- используемые инструменты;
- предпочтительный стиль обратной связи;
- предпочитаемый темп и размер задания;
- устойчивые ограничения и сложности.

Не сохраняются приветствия, случайные фразы и временные эмоции.

Агент может предлагать не более пяти обновлений за ответ. Backend принимает только разрешённые категории памяти.

### 9.2. Профиль навыков

Структурированный источник правды об уровне каждого навыка. Не хранить его как произвольные `memory_items`.

### 9.3. Контекст разговора

Сохраняются сообщения и rolling summary, но они не заменяют персональную память и профиль навыков.

## 10. Контекст агента

Каждый запрос собирается заново из:

```text
system prompt
+ персональная память
+ профиль навыков
+ состояние первичного опроса
+ активное задание с шагами и рубрикой
+ прогресс и заметки по заданию
+ rolling summary
+ последние 12 сообщений
+ новое сообщение пользователя
```

Ограничения:

- до 20 элементов персональной памяти;
- последние 12 сообщений;
- `rolling_summary` до 1500 символов;
- профиль содержит только известные навыки каталога;
- вся история не передаётся агенту;
- agent-сессия не используется как память.

## 11. Изменения модели данных

Существующие таблицы сохраняются.

Добавить таблицу:

### skill_assessments

```text
id
user_id
skill_id
self_level
assessed_level
confidence
evidence
strengths
gaps
created_at
updated_at
```

Ограничение уникальности:

```text
user_id + skill_id
```

Расширить `user_state`:

```text

```

Расширить `task_progress` при необходимости:

```text

review_summary
skill_evidence
completed_at
```

Изменения схемы должны сохранять данные существующей базы и не требовать её ручного удаления.

## 12. Структурированный результат агента

Расширить текущую схему примерно так:

```python
class MemoryUpdate(BaseModel):
    key: str
    value: str


class SkillAssessmentUpdate(BaseModel):
    skill_id: str
    self_level: str | None = None
    assessed_level: str | None = None
    confidence: str
    evidence: str
    strengths: list[str] = []
    gaps: list[str] = []


class AssistantResult(BaseModel):
    reply: str
    memory_updates: list[MemoryUpdate] = []
    skill_updates: list[SkillAssessmentUpdate] = []
    rolling_summary: str
    task_note: str | None = None
    task_completion_suggested: bool = False
    onboarding_completion_suggested: bool = False
```

Правила backend:

- пользователю отправляется только `reply`;
- неизвестные поля запрещены;
- `skill_id` должен существовать в каталоге;
- уровни и confidence валидируются по enum;
- агент не меняет `active_task_id`;
- агент не завершает опрос или задание автоматически;
- статус `done` ставится только после явного подтверждения дизайнера;
- при ошибке запуска или валидации память, профиль и прогресс не обновляются.

## 13. Telegram-сценарии

### `/start`

- создаёт пользователя;
- если опрос не завершён — начинает или продолжает его;
- если опрос завершён — показывает активное или рекомендуемое задание.

### `/tasks`

Показывает карту навыков и задания со статусами:

```text
not_started
in_progress
done
```

### Кнопки активного задания

- `Продолжить задание` — агент сразу даёт текущий или следующий шаг;
- `Завершить задание` — подтверждение завершения;
- `Выбрать другое` — переход к каталогу;
- `Объяснить подробнее` — объяснение текущего шага простыми словами;
- `Мой профиль навыков` — краткий структурированный профиль.

Свободные сообщения продолжают работать вне активного задания.

## 14. Поведение агента

Агент должен:

- быть персональным наставником, а не генератором общих советов;
- опираться на подтверждённый профиль навыков;
- задавать не больше одного важного вопроса за раз;
- давать конкретный следующий шаг;
- различать самооценку и подтверждённый уровень;
- объяснять решения простым языком;
- использовать примеры, но не подменять ими работу дизайнера;
- давать обратную связь по рубрике;
- честно обозначать недостаток данных;
- сохранять свободную смену темы.

Агент не должен:

- сообщать пользователю о JSON, внутренних схемах и техническом обновлении памяти;
- придумывать выполненную практику или доказательства уровня;
- ставить уровень `independent` только на основании уверенной самооценки;
- завершать задание без подтверждения;
- превращать каждый ответ в длинную лекцию;
- хвалить без конкретного основания.

## 15. Технические ограничения

Сохраняются:

- Python 3.12;
- aiogram 3;
- SQLAlchemy 2;
- aiosqlite и SQLite;
- Pydantic и pydantic-settings;
- pytest, pytest-asyncio, ruff;
- one-shot `codex exec` без `resume`;
- модель `gpt-5.4-mini` и `medium` reasoning effort, передаваемые приложением;
- long polling;
- доступ одному разрешённому Telegram ID.

Не строить универсальные `AgentBridge`, `ProgramEngine`, planner/evaluator framework или систему нескольких агентов. Допустимы небольшие сервисы, напрямую соответствующие профилю навыков, опросу и заданиям.

## 16. Порядок реализации v0.2

1. Расширить и провалидировать каталог навыков и заданий.
2. Добавить постоянный профиль навыков и безопасное обновление схемы SQLite.
3. Добавить первичный опрос и сохранение его состояния.
4. Включить профиль навыков и полное задание в контекст агента.
5. Обновить системный промпт под наставническое поведение.
6. Реализовать пошаговый сценарий задания и проверку по рубрике.
7. Добавить просмотр профиля и рекомендации следующего задания.
8. Обновить тесты и README.

Не очищать репозиторий и не переписывать работающий вертикальный сценарий без необходимости.

## 17. Обязательные тесты

Помимо существующих тестов:

1. Каталог содержит все 12 навыков.
2. У каждого навыка ровно одно основное задание.
3. Задание с неизвестным `skill_id` не загружается.
4. Новый пользователь начинает первичный опрос.
5. Состояние опроса сохраняется после повторной инициализации приложения.
6. После завершения опроса сохранён профиль навыков.
7. Самооценка не обновляет `assessed_level` без evidence.
8. Неизвестный навык из ответа агента отклоняется.
9. Профиль навыков попадает в prompt.
10. В prompt активного задания попадают рубрика и текущий прогресс.
11. Выбор задания не удаляет прошлый прогресс.
12. Предложение агента не завершает задание автоматически.
13. Подтверждённое завершение сохраняет evidence связанного навыка.
14. Ошибка агента не меняет память, профиль и прогресс.
15. Существующая база может быть открыта после обновления без ручного удаления.

В тестах использовать fake agent runner. Реальный агент и сетевые вызовы не запускаются.

## 18. Не входит в v0.2

- прямая интеграция с API LLM-провайдера;
- постоянная или возобновляемая agent-сессия;
- автоматическая генерация новых навыков;
- несколько заданий разной сложности на один навык;
- числовые рейтинги, баллы и достижения;
- расписание и напоминания;
- Figma, файлы, изображения и веб-поиск;
- RAG, embeddings и vector store;
- админ-панель;
- несколько пользователей или ролей;
- несколько агентов или runtime;
- PostgreSQL, Redis, Celery, FastAPI и микросервисы.

## 19. Acceptance criteria

v0.2 считается готовой, когда:

- текущий MVP продолжает запускаться и обрабатывать сообщения;
- новый пользователь проходит последовательный опрос;
- после опроса создаётся постоянный профиль всех основных навыков;
- профиль и состояние восстанавливаются после перезапуска;
- в каталоге есть 12 навыков и одно практическое задание на каждый;
- каждое задание содержит смысл, результаты, шаги, рубрику и типичные ошибки;
- после выбора задания агент сам начинает учебный сценарий;
- дизайнер может попросить подробное объяснение текущего шага;
- агент проверяет результат по рубрике и даёт конкретную доработку;
- завершённое задание обновляет evidence и уровень связанного навыка;
- свободный диалог продолжает работать;
- one-shot запуск агента и ограниченный контекст сохраняются;
- `ruff check .` и `pytest` проходят;
- merge выполняет только владелец репозитория.
